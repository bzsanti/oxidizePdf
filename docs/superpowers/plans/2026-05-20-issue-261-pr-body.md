## Summary
Fixes #261. Adds line + paragraph reconstruction to the text extraction pipeline so that `partition()`, `rag_chunks()`, and `rag_chunks_with_profile` produce Elements at paragraph granularity instead of one Element per `Tj` operator.

## Approach
Two new private methods on `TextExtractor`:
- `merge_into_lines(fragments)` — groups by baseline (Y tolerance = ½ smaller fragment height), sorts by X within each line, concatenates with spaces where `x_gap > space_threshold * font_size`.
- `merge_into_paragraphs(lines)` — groups consecutive lines whose vertical gap is within 1.5× the median line height, joining with `\n` or eliding the hyphen when `merge_hyphenated` is set and the previous line ends with `-`.

Gated by a new `ExtractionOptions::reconstruct_paragraphs: bool`, default `false` to preserve backward compatibility for direct `extract_text` callers. `partition_with` and `partition_with_profile` force it to `true`.

Public entry point `TextExtractor::merge_fragments_for_partition` exposes the full chain (kerning fix → lines → paragraphs) so the integration test can exercise the contract without going through a PDF file.

Incidental: the existing `merge_close_fragments` (kerning fix) now inserts a space between merged fragments when their X gap exceeds `space_threshold * font_size`. Previously it concatenated without a space, which conflated sub-character kerning ("IN VO ICE" → "INVOICE") with word-boundary kerning ("The first" → "Thefirst"). The new behavior keeps "INVOICE" intact while word boundaries get their space back.

## Out of scope (deliberately)
- **arXiv/TeX `Sendstream` parser bug (#260)** — separate fix.
- **TextExtractor CTM bug (#262)** — found during verification. Some PDFs (notably NCSC v4.0 of the corpus) emit `TextFragment` values whose `font_size`, `x`, `y`, and `width` are in text space rather than page space because the extractor does not consistently compose the current CTM with the text matrix. Result: on those PDFs, `font_size` may be `1.0` and inter-glyph gaps in text space (~4pt) end up larger than the relative threshold (`space_threshold * 1 = 0.3pt`), producing letter-spaced text like `"p r i n c i p l e"`. This is upstream of paragraph reconstruction and visible only because #261 makes the pipeline actually attempt geometric merging. Tracked separately.
- **Tightly letter-spaced display headings** (e.g. BSI title `"TechnischeRichtlinie–Kryptographische..."` ) still concatenate without spaces because the kerning-fix threshold `0.5 * font_size` swallows them. Tunable; logged as residual in the verification report.
- **Multi-column reconstruction** — when `detect_columns: true` is also set, the existing column detector runs before the new line merger. Composing the two correctly is a future enhancement.
- **rag_realworld example** — the downstream showcase example (PR #2 of the 3-language series) sits on `feature/rag-realworld-rust` and lands after this PR.
